### 1. The Streaming Connection Fix

The root cause of the error is an **audio encoding mismatch**.

In your Java code, the `AudioFormat` initializes the microphone to capture raw, uncompressed 16-bit PCM audio (`pcm_s16le`). However, the WebSocket connection to AssemblyAI is configured with `"encoding": "opus"`. When our servers receive raw PCM bytes but expect compressed Opus packets, the stream immediately fails.

**The Fix:** Update the encoding parameter from `"opus"` to `"pcm_s16le"` to match the actual audio format your `TargetDataLine` produces.

We also **strongly recommend** migrating to our official Java SDK and our newest model, **Universal-3 Pro** (`speech_model: "u3-rt-pro"`). Universal-3 Pro natively supports **English/Spanish code-switching** — it can handle bilingual conversations where speakers switch between languages, which is ideal for your court interpretation workflow. No special configuration needed; the model detects language transitions automatically.

> Ref: [Universal-3 Pro Streaming docs](https://www.assemblyai.com/docs/streaming/universal-3-pro)

Additionally, we noticed your code embeds the API key directly in the client application. We recommend using **temporary authentication tokens** generated by a backend server to avoid exposing your credentials.

> Ref: [Temporary auth tokens](https://www.assemblyai.com/docs/streaming#authenticate-with-a-temporary-token)

Operating at 2,000 concurrent streams is well within our platform's capabilities. Here's how it works:

**On our side:** Paid accounts start with 100 concurrent streaming sessions. Our system auto-scales: when 70% of your limit is utilized, the cap increases by 10% the following minute. **On your side:** Each WebSocket stream maintains a long-lived TCP connection. To handle 2,000 concurrent connections, ensure your own backend is horizontally scaled (e.g., behind a load balancer, with a message broker like Redis for WebSocket state).
